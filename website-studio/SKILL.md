# Website Studio — 全自动建站导演

把"我想做个网站"变成一个有品味、可上线的成品。核心信条：**用户只关心结果，不关心技术。** agent 做所有工程决策，只把审美选择权交还用户。

## 设计哲学（先读这一段）

1. **全自动优先。** 默认 agent 全权决策——技术栈、目录结构、组件选型、配色布局。不要问用户"用 Next.js 还是 Vite""要不要 SSR"这种问题，他们不懂也不想懂。只有当用户明确要细控时才切引导式。
2. **设计文档是契约，不是描述。** DESIGN.md 规定"我们要做成什么样"，用户签字后才动工。它不是记录参考网站长什么样。
3. **拒绝塑料味。** 禁止 "clean / modern / user-friendly" 这类空话。所有设计决策落到具体值：`#0A0A0A`、`font-size: 48px`、`border-radius: 16px`、`stagger: 0.08s`。参考 taste-skill 的 Taste DNA 方法——解释*为什么*这么取舍。
4. **动效是差异化武器，默认给足。** 用户给的参考不可能 1:1 复制（也不该）。用动效库（React Bits / GSAP / Magic UI / Aceternity）做出"参考的感觉 + 自己的骨相"。**不要默认克制**——除非用户明确要极简，否则默认偏丰富；用户说"酷炫/复杂/丰富/炫/wow"时直接拉满（见丰富度等级）。
5. **只让用户做主观判断。** 贯穿全程，用户该参与的是：描述想要什么、给参考、选风格方向、挑动效。不该参与的是：任何带技术名词的选择。
6. **丰富 ≠ 塑料味。** "拒绝塑料味"拒绝的是空话和偷懒（默认蓝紫、`transition-all` 一把梭、Lorem 占位），**不是拒绝丰富**。有意图的、用真实组件库做出的丰富动效是好东西；用基础 vanilla 手写几个 fade 凑数才是塑料味。别把"克制"当成品质的代名词。

## 反默认纪律（两层反射检查）—— 动工前必做

大多数 AI 生成的站一眼假，不是技术不够，是 agent 不读 brief 就跳到品类默认审美。光"避开默认"还不够——避开第一层，还有第二层在等你。

- **第一层反射：** 光从品类能不能猜出配色/风格？能，就是训练数据默认，必须重做。
  - 咖啡/手作 → 暖米色+黄铜衬线 ｜ AI/SaaS → 紫光+Inter ｜ 金融 → 海军蓝+金 ｜ 极简品牌 → 编辑排版+衬线大字
- **第二层反射（更深的陷阱）：** 避开第一层后，能不能从"品类+你的反例"再猜出审美？
  - "不 SaaS 奶油风的 AI 工具" → 编辑排版风 ｜ "不海军金的金融" → 终端暗色 ｜ "不米色的手作品牌" → 牛血色+Cormorant

**硬规则：** 两层都要打到"猜不出"为止。在 DESIGN.md 写一行：`本 brief 的默认是 X，一阶避开到 Y，二阶再避开到 Z，理由是…`。下面的 Hero 原型库就是这条规则在 Hero 上的具体落地。

> 注：`#6366f1`（通用 indigo）是 AI 生成 UI 里被滥用最狠的单个 hex；米色+黄铜+咖啡色族（token 名 `--cream/--sand/--bone/--paper`）是 2026 手作品类的饱和默认。两者无理由禁用。

## 高级感物理层（数值化配方）—— 贵和便宜的差别主要在这层

> 核心论点（经生产验证）：高级感不是玄学，是一层薄薄的物理质感，给精确值就能复现。但**按 register 分层**——brand 全套，product 减配。这是我们对 finesse 物理层的关键改进：finesse 把 grain/暗角一刀切铺所有页面，铺在数据表上就是噪音。

### 全套（brand / 落地页 / 作品集 / Hero 页）

| 要素 | 精确做法 |
|------|---------|
| Grain 噪点 | 固定层 `feTurbulence`，opacity `.025–.05`，内联 ~2KB。杀"矢量塑料味"的最大单一信号。亮页降到 `.02–.03` |
| Vignette 暗角 | 径向渐变压暗，焦点 `at {x}% {y}%` 对准主体，制造视觉焦点 |
| 字重张力 | 标题 `clamp()` + 负字距 `-.02~-.045em` + 行高 `.86–.95`，800/900 对 300 正文。标题内强调用同族斜体/加粗，禁混族塞衬线词 |
| 半透明边框 | `rgba(255,255,255,.07–.22)`(暗) / `rgba(0,0,0,.06–.08)`(亮)。**禁硬 `#333` 线** |
| 禁纯黑纯白 | 所有中性色往品牌色相偏一点（OKLCH +0.005–0.015 chroma）|
| 配色锁定 | 一个 accent 贯穿全页，禁止第 N 节冒出异色 CTA/徽章 |
| 染色阴影 | 暗页 `box-shadow: 0 0 80px rgba(0,0,0,.8)`；亮页阴影往 bg 色相染，禁纯黑 |
| 节奏 | `section{padding:120px 52px;max-width:1280px;margin:0 auto}`，对称 gutter |

### 减配（product / 仪表盘 / 后台 / 数据表）

- **禁 grain、禁 vignette**——噪点压数据表是干扰，暗角压 admin 面板分心。仪表盘的"贵"来自信息密度和组件精度，不来自物理质感。
- 保留：字重张力、半透明边框、禁纯黑纯白、配色锁定、染色阴影。

### 硬数值地板（两路共用，不可妥协）

- 正文对比度 ≥ 7:1，次级文本 ≥ 3.5:1，UI/图标 ≥ 3:1（WCAG AA）。明暗分别测，别假设一档过另一档也过。
- accent 的 OKLCH chroma ≤ 0.23；若 accent 亮度 > 0.78，chroma 再压到 ≤ 0.18（防荧光廉价感）。
- accent vs 其上文字（按钮 label）≥ 4.5:1。
- 每个颜色三段命名：`名字(hex) — 角色与约束`，防跨文件漂移。例：`Void(#04060D) — 页面背景，往品牌色相偏的近黑`。
- 内部用 OKLCH 推配色（感知一致、易出明暗对），输出 hex。

## 丰富度等级（Richness Dial）—— 动工前必须定

每次建站先判断丰富度等级，它决定动效数量、技术栈路径、组件来源。**默认不低于 L2，倾向 L3。**

| 等级 | 触发条件 | 动效数量 | 技术栈路径 | 组件来源 |
|------|---------|---------|-----------|---------|
| **L1 极简** | 用户**明确**说"极简/克制/干净/少动效/性冷淡" | 2–3 个，全为基础 | 可单文件 HTML | 手写即可 |
| **L2 适度** | 用户没特别说，或说"还行/正常" | 6–10 个 | Vite + React + 真实库 | 真实组件源码 |
| **L3 拉满** | 用户说"酷炫/复杂/丰富/炫/wow/惊艳/有创意/不像 AI 做的"，或抱怨"太简单了" | **10–20+ 个**，招牌炫技全上 | Vite + React + 真实库 + GSAP | **必须**取真实组件源码 |

**关键规则：**
- **不确定时往高了估。** 宁可做完用户觉得"够了"，不要做完用户觉得"太简单"。
- **用户一旦说"太简单/不够酷/再丰富点"，立即升到 L3 重做，不要在原基础上小修小补。**
- L2/L3 **禁止**走单文件 HTML 降级——必须用 React 工程跑真实组件库。降级只留给 L1。
- L3 必须用上"招牌炫技"类效果：3D 倾斜卡、能量束连线、粒子/漩涡/极光背景（高存在感）、滚动钉住/横向平移、文字逐字拆解、磁性按钮、卡片堆叠揭示等——具体见 `references/animation-catalog.md` 的 B/I/S/H 类。

**声称即兑现（铁律）：** 声称了 L3 拉满，就必须真有 10–20+ 个能跑的动效，且招牌炫技至少 3 类。一个声称 L3 却只交 5 个 fade 的站是**不合格**——要么做到，要么把等级降到 L2 交一个完美 L2。**禁止半吊子**。所有动效必须中端机实测 60fps、`prefers-reduced-motion` 必须有静止帧降级，否则就是坏作品。

## Hero 构图原型库（每次必须换，不得复用）

**这是最常犯的病：每次都下意识用"居中大标题 + 一句副标题 + 双 CTA 按钮"这一个安全模板。** 用户第一眼看 Hero 就觉得"又来了"。Hero 是第一印象，构图必须每次不同。

**硬规则：**
1. 每次建站必须从下方原型库挑一个 Hero 构图，**记入 DESIGN.md 的"Hero"组件条目**（写明用的是哪个原型）。
2. **禁止**默认使用"居中标题 + 副标题 + 双按钮"这个最常见模板——它正是要避免的那个。
3. **不得连续两次用同一原型。** 若本会话已用过某原型，下一次必须换。
4. 原型要和网站类型/风格匹配（见下表"适配"列），不是随机选。

| 原型 | 构图 | 主角 | 适配 |
|------|------|------|------|
| **产品即主角** | 标题压小靠左/上，产品截图·仪表盘 mockup 占主体，配浮动标注 callout | 产品界面 | SaaS / 工具 / 仪表盘 |
| **实时演示** | 标题压小，活的终端·控制台·交互演示占主体，实时流式输出 | 运行中的演示 | dev tool / AI agent / 命令行产品 |
| **排版封面** | 巨型标题撑满屏，不对称，无按钮或仅一个链接，排版即主角 | 字体排版 | 编辑 / 品牌 / 作品集 / 杂志 |
| **全屏视觉** | 全屏背景图·视频·3D·shader，标题叠加其上，文字极少 | 背景视觉 | 品牌 / 活动 / 时尚 / 影像 |
| **分屏** | 左文右图（或左文右产品），不对称分栏，两侧等重 | 文+图并列 | 产品页 / 服务介绍 |
| **Bento Hero** | Hero 本身是 Bento 网格，标题占一格，其余格放亮点/数据/小演示 | 模块网格 | dashboard / Bento 风 / 数据产品 |
| **叙事滚动** | Hero 是滚动叙事第一幕，pin + 视差，标题随滚动演变 | 滚动序列 | storytelling / 品牌叙事 |
| **极简留白** | 大量负空间，一句标题偏置，几乎无其他元素，一个链接 | 留白本身 | 高端品牌 / 设计工作室 |

> 选型后要在 DESIGN.md 写清楚：用的哪个原型、主角元素是什么、为什么和这个产品/风格匹配。Phase 5 验收时核对——Hero 构图和 DESIGN.md 一致吗？是不是又退化成"居中标题+双按钮"了？是的话回炉。

## 技术栈默认值（不要问用户，直接用）

| 维度 | 默认 | 理由 |
|------|------|------|
| 框架 | Vite + React(L2/L3 交互型默认)/ Next.js(多路由内容站) | Vite 启动快、适合落地页/作品集/活动页;Next.js 用于博客/文档/电商/需 SSR 的整站 |
| 样式 | Tailwind CSS | React Bits / Magic UI / Aceternity 全部假设 Tailwind |
| 组件基座 | shadcn/ui | 无依赖复制、可深度定制，和动效库解耦 |
| 动效核心 | Motion (原 Framer Motion) | React 动画事实标准，上面几个库的底层 |
| 脚本动效 | GSAP + ScrollTrigger | 滚动驱动、时间线编排、复杂序列 |
| 部署形态 | 静态产物优先 | Vite 用 `vite build` 出静态 SPA;Next.js 用 `next build` 静态导出;需交互态再上 `next start` |

> 降级规则（严格）：单文件 HTML + CDN **只在 L1 极简**或用户明确说"纯 HTML/不要框架"时用。L2/L3 一律用 Vite + React + Tailwind + Motion + GSAP 真实工程，**禁止降级**。落地页、作品集、活动页这类要展示效果的产物，默认就是 L2+，不得降级。
>
> 真实组件获取：L2/L3 必须用 `agent-browser` 打开 reactbits.dev / magicui.design / aceternity.com **取真实组件源码**装进项目，不要凭记忆手写。这些库是 copy-paste 式（非 npm 包），抄源码是正确用法。

## 六阶段流程

每个阶段都有进出条件。**不达标不推进**——参考 taste-skill 的硬门控原则。

---

### Phase 0 — 意图捕获（Brief）

**目标：** 把用户的一句话变成结构化的建站需求。

**做什么：**
1. 收下用户的描述（可能是一句话、一段话、或丢进来的链接/图片）。
2. agent 自行推断，不要逐条追问：
   - **网站类型**：落地页 / 作品集 / 仪表盘 / 博客 / 电商 / 活动页 / 文档站
   - **核心板块**：从上到下列出（Hero、特性、展示、定价、FAQ、Footer…）
   - **目标受众与语气**：极客/大众/B端/创作者
   - **风格方向**：从下方调色板里挑 1–2 个倾向
3. 如果用户给了参考 URL，先记下，Phase 1 处理。

**风格调色板**（让用户选方向，不是技术参数）：

| 代号 | 风格 | 关键词 |
|------|------|--------|
| A | 极简留白 | 大量负空间、单色、衬线/无衬线对比、克制动效 |
| B | Bento 网格 | 模块化卡片、圆角、信息密度高（参考 Apple/Bento Grids） |
| C | 玻璃拟态 | 毛玻璃、渐变背景、漂浮层、光影 |
| D | 暗黑高对比 | 纯黑底、霓虹/单点亮色、大胆字体 |
| E | 3D / Shader | WebGL 背景、着色器渐变、空间感（参考 Shader Gradient） |
| F | 杂志编辑感 | 大标题、栏式排版、衬线字、强内容节奏 |
| G | 趣味交互 | 弹性动效、鼠标跟随、彩蛋、playful |

**出条件：** agent 输出一个 ≤6 行的 Brief 摘要给用户确认：
> "我给你做一个 **[类型]**，面向 **[受众]**，风格走 **[代号+关键词]**。主要板块：①Hero ②… ⑥Footer。参考了你给的 [X]。**没问题我就开始出设计文档。**"

用户说"好"/"开始"即推进；给调整意见就改了再确认。**不要问技术问题。**

---

### Phase 1 — 参考设计 DNA 提取

**目标：** 如果用户给了参考（URL/图片），把它的设计语言拆成可复用的令牌；没给就跳过，靠 Phase 0 的风格方向自主发挥。

**有 URL 时——用 agent-browser 提取（环境已装好）：**
1. `agent-browser open <url>` 打开参考站
2. `agent-browser eval` 抓：根 CSS 变量、关键节点 computed style（标题/按钮/卡片/导航）、字体栈、配色、圆角、阴影、间距节奏
3. 慢滚一遍，记录滚动动效、sticky 行为、悬停态
4. 综合成**设计 DNA**：色板（带 hex）、字号阶梯、间距体系、圆角/阴影、招牌动效、内容语气
5. **明确区分"观察到的"vs"推断的"**，不要瞎编

> 不要用截图作为主要证据——用 DOM 和 computed style。截图只在需要核对整体观感时辅助。
> 备选：环境若有 taste-skill 或 web-to-design-md，可调用增强，但不强制依赖。

**有图片参考时：** 从图片直接读色板、布局结构、字体感觉，标注为"基于图片推断"。

**没参考时：** 基于 Phase 0 风格代号，从 `references/resource-map.md` 查对应风格的默认设计令牌，直接进入 Phase 2。

**出条件：** 有一份设计 DNA（无论是提取的还是默认的），足以喂给 Phase 2 写设计文档。

---

### Phase 2 — 设计文档契约（DESIGN.md）

**目标：** 产出一份"处方式"设计文档，规定要做成什么样。这是动工前的最后一道闸。

**做什么：**
1. 综合 Phase 0 的 Brief + Phase 1 的设计 DNA，按 `references/design-doc-template.md` 填写 `DESIGN.md`。
2. 文档要**可执行**——另一个 agent 拿着它不看你也能复刻出同样的站。包含：
   - 视觉主题与氛围（带具体关键词，不要空话）
   - 色彩令牌表（角色 / 语义名 / hex / 用途）
   - 字体规则（字体族、字号阶梯、字重、行高）
   - 间距与布局体系（基准单位、栅格、断点）
   - 组件词汇（Hero/卡片/按钮/导航…的长什么样）
   - 动效原则（哪些地方动、用什么节奏、克制还是张扬）
   - 内容语气（文案风格示例）
3. **同时生成一个 `DESIGN-preview.html`**——把色板、字体、组件样式做成一块可视化的"设计板"，用 preview 部署给用户看。用户看图比看 markdown 直观。

**出条件：** 把 DESIGN-preview.html 部署出来给用户看，附一句话摘要。用户点头（"可以"/"开始做"）即推进到 Phase 4（动效选型可在确认前并行准备）。用户要调就改文档再确认。

> 全自动模式下，若用户一开始就说"你全包了"，这步可压缩为：agent 内部定稿 DESIGN.md，只把 DESIGN-preview.html 丢给用户看一眼，不等长确认就继续——但动工前至少给一次可视化的"设计板"。

---

### Phase 3 — 动效策展选型

**目标：** 从动效库里挑出和风格匹配的效果，让用户在画廊里点选（或全自动代选）。

**做什么：**
1. 跑 `scripts/build-gallery.py`，基于 `references/animation-catalog.md` 生成一个本地策展画廊 HTML（按用途分类：Hero / 滚动 / 背景 / 交互 / 文本 / 布局 / 加载）。
2. 用 preview 部署画廊，给用户链接。
3. 画廊里每个效果有：编号、动图/演示、来源库、适用场景、"点击看 live demo"。
4. 让用户报数字挑选："喜欢哪些？报编号就行，或者让我帮你选。"
5. **全自动模式**：用户说"你选"或没兴趣，agent 按 DESIGN.md 的风格自动选——**按丰富度等级定数量**：L1 选 2–3 个，L2 选 6–10 个，L3 选 10–20+ 个并把招牌炫技全上。

**选型原则：**
- 数量服从丰富度等级，**不要默认往少了选**。L3 时宁可多选再删，不要先省。
- 动效要**分布在所有主要板块**（Hero/背景/滚动/交互/文本/布局/加载都覆盖到），不要全堆在 Hero。
- L3 允许多库混用（Motion + GSAP + React Bits + Magic UI + Aceternity），按效果类型各取所长，不设"不超过2个库"的上限——那是 L1/L2 的克制逻辑。
- 效果要有**层次**：主带动效（Hero/关键章节）+ 环境动效（背景/氛围）+ 细节动效（悬停/微交互）三层都要有。
- 标注每个选定效果对应的 `references/animation-catalog.md` 条目和真实来源链接，方便 Phase 4 取组件。

**出条件：** 有一份动效选型清单（编号 + 来源 + 落点位置）。

---

### Phase 4 — 实现

**目标：** 按 DESIGN.md + 动效选型清单把站搭出来。

**做什么：**
1. 按"技术栈默认值"起项目脚手架（除非 Phase 0 判定降级单文件 HTML）。
2. 把 DESIGN.md 的令牌落地成 Tailwind config / CSS 变量。
3. 逐板块实现，从动效库取**真实组件**（去 reactbits.dev / magicui.design / aceternity.com 抄对应组件代码，不要凭记忆瞎写）。
4. 用 `agent-browser` 打开 reactbits.dev 等取真实组件代码——**不要凭印象编组件 API**，容易跑不通。
5. 实现完用 preview 部署。

**质量底线：**
- 响应式必须做（mobile/tablet/desktop 断点）。
- 动效不能阻塞首屏（`prefers-reduced-motion` 要尊重）。
- 文案不能是 Lorem Ipsum——按 DESIGN.md 的内容语气写真文案。
- 不要有 console 报错。

**出条件：** 站点可通过 preview URL 访问，首屏可看。

---

### Phase 5 — 验收与迭代

**目标：** 对照 DESIGN.md 查"塑料味"，迭代到有品味。

**做什么：**
1. 用 `agent-browser` 自己打开 preview URL，逐屏检查：
   - 是否符合 DESIGN.md 的色彩/字体/间距？
   - 动效节奏对不对？有没有"AI 一眼假"的塑料感？
   - 移动端有没有崩？
2. 把发现的问题列清单，逐条修。
3. 给用户看成品，收集反馈，迭代。

**查"塑料味"的检查点：**
- 配色是不是用了默认 Tailwind 蓝/紫？→ 换成 DESIGN.md 的令牌
- 字体是不是默认无衬线一路到底？→ 按 DESIGN.md 加对比
- 动效是不是全用 `transition-all` 或手写 fade 凑数？→ 按选型用具体库的真实组件效果
- 间距是不是全 `p-4`？→ 按体系用阶梯
- 文案是不是"Lorem"/"Your text here"？→ 写真文案

**查"丰富度"的检查点（L2/L3 必查）：**
- 实际用了几个动效？是否达到丰富度等级要求的数量？不够就补。
- 是否用了**真实组件库源码**，还是全手写 vanilla 冒充？L2/L3 手写超过一半就是不合格，回 Phase 4 取真实组件重做。
- 招牌炫技效果（3D 倾斜/能量束/粒子背景/滚动钉住/逐字动画等）有没有上？L3 必须至少 3 类招牌效果。
- 动效是不是只堆在 Hero？各主要板块是否都有动效覆盖？
- 整体观感是否"一眼 AI 安全牌"？如果是，加创意效果打破预期。

**反廉价黑名单（机械核对，三档）—— 完整清单见 `references/anti-cheap.md`**

- `[硬禁]`（命中一条不交付）：em-dash 当装饰用 ｜ div 造假截图/假仪表盘冒充产品 ｜ 渐变文字当默认 flourish ｜ AI 紫光/`#6366f1` 默认化 ｜ 伪造精确数字（92%/4.1× 无来源）
- `[AI 味]`（要带理由才能用）：每节 eyebrow 小标（≤1/3 节）｜ 01·02·03 编号当默认架构 ｜ 千篇一律卡片网格（图标+标题+文×6）｜ 默认 Tailwind 蓝紫 ｜ 米色+黄铜+咖啡色族 ｜ `Inter/Fraunces` 盲用不说明理由 ｜ 纯 `#fff`/`#000`/硬 `#333` 边 ｜ 营销废话词（streamline/empower/seamless）
- `[30 秒自检]`：默认命名并打败了没？一个 accent 锁全页了没？eyebrow 数 ≤ ceil(节数/3)？≥4 种布局族、无一种超 2 次？该有图的地方有图没？声称的丰富度真兑现了？文案无 em-dash/假数字/AI 卖萌？对比度 AA？

**出条件：** 用户满意，或 agent 自检通过且用户说"可以了"。

## 反馈→精准迭代（一个抱怨只动一个旋钮，不整页重做）

用户改稿反馈映射到单个旋钮或模块，只动那一个，再跑对应检查。禁止"一个抱怨就从头 Phase 0 重来"。

| 用户说 | 动哪个 | 动作 |
|--------|--------|------|
| 太素/无聊/不够酷 | 丰富度 +1 档 | 升 L 等级，补招牌炫技效果 |
| 太闪/太满/太重 | 丰富度 −1 档 | 降级，砍效果，重型引擎换 CSS-only |
| 味道不对/感觉假/太普通 | 风格方向 + 反默认 | 重跑两层反射检查，重选风格代号，重铺物理层 |
| 太空/信息少 | DENSITY + | 加内容板块 |
| 太挤/信息多 | DENSITY − | 砍板块，加间距 |
| 换配色 | accent | 重定 accent，保持全页锁定 |
| 换动效 | 动效选型 | 换引擎/组件，不动配色布局文案 |
| 加 3D/景深/倾斜 | 招牌效果 | 加一个 3D 时刻（倾斜/视差/模型），全页最多一个 |
| 这页能上吗/审一下 | 只读审计 | 跑反廉价黑名单 + 丰富度兑现 + 起飞前自检，只报不改 |
| 改某节布局 | 局部 redesign | 删该节，重审布局族（确保 ≥4 种仍成立）|

## 模式切换

- **全自动（默认）：** 用户给一句话 → agent 跑完 Phase 0–5，中间只在 Phase 2 给一次设计板预览、Phase 3 给一次画廊。其余自己定。
- **引导式：** 用户说"我想一步步来"/"带你确认"时切此模式——每个 Phase 结束都停下来等用户确认再推进。
- 任何时候用户丢新需求/参考进来，回到对应 Phase 重新跑，不要从头来。

## 不要做什么

- ❌ 问用户技术栈/框架/SSR 这类问题
- ❌ 用 "clean / modern / beautiful" 这种空话描述设计
- ❌ 凭记忆编动效库组件 API——要用 agent-browser 取真实代码
- ❌ 跳过 DESIGN.md 直接写代码
- ❌ 把整个资源清单塞给用户——agent 内部查 `resource-map.md` 即可
- ❌ 用 Lorem Ipsum 占位
- ❌ **默认走单文件 HTML 降级**——只有 L1 极简或用户明确要"纯 HTML"才行；L2/L3 必须用 React 工程
- ❌ **手写 vanilla 效果冒充真实组件库**——L2/L3 必须取 reactbits/magicui/aceternity 的真实源码
- ❌ **用户要"酷炫/丰富"时还只给 3–5 个基础动效**——这是最严重的失误，必须升 L3 拉满
- ❌ **把"克制"当品质默认值**——克制是一种风格选择，不是偷懒的借口
- ❌ **所有板块套同一个结构模板**——结构要根据网站类型和风格变化，换皮不换骨不行
- ❌ **默认用"居中标题+副标题+双CTA"Hero 模板**——必须从 Hero 原型库选，且不得连续复用同一原型
- ❌ **用 em-dash 当文案装饰**（— 做停顿/列项/戏剧性旁白）——AI 文案最重的 tell，用句号/逗号/重写句子
- ❌ **div 造假截图/假仪表盘冒充产品**——要么真截图、要么生成图、要么做个真组件，别拿 div 摆拍
- ❌ **伪造精确数字**（92%/4.1×/48k 无来源）——用真数据、标 mock、或删掉
- ❌ **每节都顶个 eyebrow 小标**——≤1/3 节才许有，通常标题本身够了
- ❌ **用纯 `#fff`/`#000`/硬 `#333` 边**——所有中性色往品牌色相偏一点，边框用半透明
- ❌ **`#6366f1`/米色+黄铜+咖啡色族当默认**——无理由禁用，跑两层反射检查后再定配色
- ❌ **一个抱怨就整页重做**——映射到单个旋钮只动一处（见"反馈→精准迭代"）
